# 围猎模式：阶段一

用于"漏一个就失败"级别的完备性需求（安全审计、迁移评估、合规检查等）。通过多个独立专家的并行收集和对抗性审查，突破单人注意力预算的限制。

你的工作分两个阶段，**分步加载、分步执行**：

1. **理解问题**（本文件）——理解对齐 + 多专家并行收集
2. **分析与呈现**（`phase-2-multi.md`）——推理、合成、呈现、迭代

**在阶段一完成并获得用户确认之前，不要读取 `phase-2-multi.md`，不要做任何分析和结论。**

---

## 工作目录

进入围猎模式后，立即在 skill 目录下创建本次分析的工作目录：

```
sessions/{YYYYMMDD-HHmmss}/
```

（即本文件所在目录的 `sessions/` 子目录）

所有子 agent 的产出写入此目录下的文件。编排者通过文件协调 agent 之间的信息传递，不在自己的上下文中持有完整产出——只持有摘要级信息做流程决策。

## 步骤 1：理解与对齐（NC1 + NC2）

**熟悉环境。** 在回应用户之前，先快速了解你所在的环境——项目是什么？已有什么？用户在什么语境下提问？

**对齐问题。** 确认终态和动机——为什么做、做成什么样、改造已有的还是从零开始。问题清晰时，用自己的话复述让用户确认。问题模糊时，列出你能想到的几种理解方向，让用户选择。

## 步骤 2：侦察（NC3 前置）

拆解的质量受限于编排者对问题域的认知。多轴交叉能对抗**思维盲点**，但对抗不了**信息盲区**——不知道存在的东西，无论怎么交叉都拆不出来。

在拆解之前，spawn 一个轻量的侦察 agent（`agents/scout.md`），任务是**绘制地图，而非深挖发现**：

- 传入：用户的问题描述、对齐后的理解、工作目录
- 产出写入 `terrain-map.md`

侦察者回答三个问题：
1. **涉及哪些域？** — 模块、系统、概念域的清单
2. **各域的规模和复杂度？** — 粗粒度评估（几个文件 vs 几十个 vs 几百个；逻辑简单 vs 交织复杂）
3. **有没有预期之外的关联？** — 编排者可能没想到但实际存在的跨域依赖

**侦察不是收集。** 侦察者不需要深入理解代码逻辑或回答子问题，只需要告诉编排者"哪里有东西值得看"。停止条件：一旦形成对各域规模和跨域关联的粗粒度地图就结束，不深入任何单域——深入是下一阶段收集者的工作。

编排者读取 `terrain-map.md`，结合用户的问题理解，进入步骤 3。

## 步骤 3：拆解与分类

拆解定义了后续收集的搜索空间——如果这里漏了一个维度，后面收集做得再彻底也补不回来。

### 3a. 拆解

用适合问题的方式拆解，选择时说明理由。不同问题有不同的自然结构（假设空间、矩阵、层级树、因果链等——参见方法论）。

### 3b. 以地图验拆解（硬性关卡）

侦察者的地形图是独立于你的拆解的事实基础。用**三路交叉**验证完备性：

1. **地图 → 拆解**：地形图中的每个域和每条跨域关联，在拆解中都有归属吗？没有归属的是候选盲点。跨域问题最容易被遗漏。
2. **独立推导**：不看地形图也不看自己的拆解，仅从问题描述出发推导"应该覆盖哪些方面"。
3. **取并集**：三份清单（拆解、地形图、独立推导）的并集才是最终的子问题空间。对只出现在一份清单中的项，判断是"确实不涉及"还是"被遗漏了"。

### 3c. 分类

对拆解出的每个子问题，判断它是收敛型还是发散型。

### 3d. 呈现

将拆解结构、三路交叉验证结果和子问题清单呈现给用户确认后，进入步骤 4。

## 步骤 4：多 Agent 收集（NC3）

spawn 子 agent 时，读取对应的 `agents/` 下的 prompt 文件作为子 agent 的指令传入。不要将 agent prompt 读入编排者自己的上下文。

### 4a. 规划收集 agent

每个子问题组 spawn **两个**收集 agent——自底向上和自顶向下各一个。两个方向有系统性的互补盲点（A/B 测试实证），放在同一个 agent 里会因注意力锚定而丧失互补性。

子问题分组原则（决定哪些子问题合并给同一对 agent）：
- **拆分条件**（两条都满足才拆为独立的一对 agent）：范围足够聚焦、所需信息源与其他子问题重叠少
- **合并信号**：多个子问题共享同一批信息源，合并为一组避免重复发现

向用户确认分组方案。

**数量默认公式**：agent 数 = 子问题组数 N × 2（方向）+ 1（缺口扫描）。

边界调节：
- N=1 且问题不复杂时，可省缺口扫描（bottom-up + top-down 的互补性已足够对抗盲点）
- N>4 时，先审视能否合并相近子问题组——过多独立 agent 会让合并阶段的噪声超过收益

### 4b. 并行收集

同时 spawn 所有收集 agent，每个传入：问题描述、该 agent 负责的子问题（含类型标注）、工作目录。

- 自底向上 agent 使用 `agents/collector-bottom-up.md`，产出写入 `findings-{组名}-bottom-up.md`
- 自顶向下 agent 使用 `agents/collector-top-down.md`，产出写入 `findings-{组名}-top-down.md`

同一子问题的两个 agent 互不可见，不同子问题的 agent 也互不可见。

### 4c. 合并与去重

所有收集者完成后，编排者读取所有 `findings-*.md` 文件，合并写入 `findings-merged.md`：
- 去重（同一事实被多个 agent 发现）
- 标注来源（哪个 agent 发现的、哪个方向）
- 仅单方向发现的项特别值得关注——可能是另一方向的盲点

### 4d. 缺口扫描

Spawn 缺口扫描者（`agents/gap-scanner.md`），传入：问题描述、完整的子问题清单、`findings-merged.md` 的路径。

缺口扫描者读取 `findings-merged.md`，产出写入 `gaps.md`。

**不传收集者的搜索过程。** 缺口扫描者只看到合并后的发现，不知道发现是怎么来的。

### 4e. 最终合并

编排者读取 `gaps.md`，将补充发现合并到 `findings-merged.md`（或追加为 `findings-final.md`）。

## 步骤 5：呈现认知基础，请求确认

编排者读取 `findings-final.md`，提炼摘要呈现给用户：

1. **问题陈述** — 用自己的话重述，包括你判断的子问题类型
2. **已掌握的关键信息**（按子问题组织，标注来源）
3. **拟定的分析方向**
4. **需要你帮忙的地方**（**大多数情况下不需要此模块——没有就不写。** 仅当穷尽代码库搜索、git 历史、网络搜索后，仍有只能由用户提供的关键信息时才出现。说清：需要什么、已尝试了哪些手段、为什么只有用户能提供）

请求用户确认后，读取 `phase-2-multi.md` 进入阶段二。
