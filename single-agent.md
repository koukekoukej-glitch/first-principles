# 深潜模式：阶段一

进入后立即在 skill 目录下创建工作目录 `sessions/{YYYYMMDD-HHmmss}/`（即本文件所在目录的 `sessions/` 子目录），用于存放分析产出和最终报告。

你的工作分两个阶段，**分步加载、分步执行**：

1. **理解问题**（本文件）——搞清楚状况，建立扎实的认知基础
2. **分析与呈现**（`phase-2-single.md`）——推理、合成、呈现、迭代

**在阶段一完成并获得用户确认之前，不要读取 `phase-2-single.md`，不要做任何分析和结论。**

---

## 步骤 1：理解与对齐（NC1 + NC2）

**熟悉环境。** 在回应用户之前，先快速了解你所在的环境——项目是什么？已有什么？用户在什么语境下提问？

**对齐问题。** 确认终态和动机——为什么做、做成什么样、改造已有的还是从零开始。问题清晰时，用自己的话复述让用户确认。问题模糊时，列出你能想到的几种理解方向，让用户选择。

## 步骤 2：拆解与分类

拆解定义了收集的搜索空间——漏一个维度，后面再怎么穷举也补不回来。

### 2a. 拆解

用适合问题的方式拆解，选择时说明理由。不同问题有不同的自然结构：

- 诊断/排错 → 假设空间（候选原因列表）
- 系统评估/审计 → 按纵深的层级清单、或维度交叉矩阵
- 方案设计 → 约束清单 + 利益方关注点
- 层级系统 → 树状分解
- 因果分析 → 因果链 / 依赖图

### 2b. 独立推导对比（硬性关卡）

Spawn 一个干净上下文的子 agent，**仅传入原始问题描述**（不传入你的拆解结果），任务是独立列出："要完整解决这个问题，应该覆盖哪些方面？" 产出写入工作目录下的 `independent-checklist.md`。

你读取该文件，对比自己的拆解：
- 仅在子 agent 清单中的 → 候选盲点，判断是"确实不涉及"还是"被遗漏了"
- 仅在自己拆解中的 → 你独有的视角，通常保留

**为什么必须用子 agent 而非自己重推一遍**：LLM 在同一上下文内无法真正"忽略"已写过的内容——自己重推会被自己的拆解锚定，形式上独立、实质上同质。必须用干净注意力池的子 agent，这一步才不是形式主义。

### 2c. 分类

对拆解出的每个子问题，判断它是收敛型还是发散型。

将拆解结构、完备性检查结果和子问题清单呈现给用户确认后，进入步骤 3。

## 步骤 3：收集（NC3）

对每个子问题，**依次**从两个方向收集，不要混在一起做。

**"穷举"的判据不是主观感觉，而是步骤 3c 的独立推导对比——只有独立推导清单的每一项都被覆盖或被明确判定为"确实不涉及"，才算完成。在 3c 验证前，不要宣告"收集完成"。**

### 3a. 自底向上收集

从已有的具体材料出发，逐层扫描：
1. **直接相关材料**：与子问题直接相关的文件、数据、记录、实现——它们的内容、结构、依赖关系
2. **关联材料**：被直接材料引用或依赖的其他材料，以及引用直接材料的上游
3. **历史与变更**：这些材料的变更历史、版本演进、过去的决策记录

### 3b. 自顶向下收集

从问题领域的通用知识出发，逐层映射：
1. **知识框架**：这类子问题通常涉及哪些维度？常见的坑在哪？如需要可搜索外部资料
2. **逐项映射**：知识框架的每一项 → 在当前情境中是什么情况？用工具验证，不靠猜
3. **反向扫描**：当前情境中有没有知识框架没覆盖到的特殊情况？

**类型决定深度**：
- 收敛型子问题：两个方向都要**穷举**，不要在"够用"时停止
- 发散型子问题：自底向上重点收集**约束和边界**，自顶向下重点收集**先例和素材**

### 3c. 自检缺口

所有子问题收集完成后，退一步审视：
1. 复用 2b 产出的 `independent-checklist.md`——它列出了"要完整解决这个问题应该覆盖哪些方面"。现在这些方面每一项都有对应证据被收集了吗？
2. 额外扫一遍：收集过程中是否浮现了 2b 子 agent 还没意识到的新维度？有则补入清单
3. 对候选缺口，用工具验证并补充

不要再 spawn 一次独立推导子 agent——那会与 2b 重复。3c 的任务是**从发现端**（已收集到什么 vs 清单说应该有什么）做缺口审计，不是从问题端重新推一遍。

收集过程中遇到关键信息及时告知用户——避免用户长时间感知不到进度、失去方向感。

## 步骤 4：呈现认知基础，请求确认

1. **问题陈述** — 用自己的话重述，包括你判断的子问题类型
2. **已掌握的关键信息**（附来源）
3. **拟定的分析方向**
4. **需要你帮忙的地方**（**大多数情况下不需要此模块——没有就不写。** 仅当穷尽代码库搜索、git 历史、网络搜索后，仍有只能由用户提供的关键信息时才出现。说清：需要什么、已尝试了哪些手段、为什么只有用户能提供）

请求用户确认后，读取 `phase-2-single.md` 进入阶段二。
