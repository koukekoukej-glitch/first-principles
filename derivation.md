# First Principles Skill：正向推导链

**文档定位**：本文档是 skill 的**设计留底 / ADR**，面向人类维护者——解释每条指令为什么存在、如何从根问题推导得来。

**不要在运行时加载本文件。** skill 运行所需的全部知识已经分布在 SKILL.md、single-agent.md、multi-agent.md、phase-2-*.md、presentation.md 中。本文档的价值是：未来修改 skill 时，能追溯到每条约束的推导源头，不会误拆关键设计。

---

本文档是 skill 所有指令的理论根基。每条指令都能沿推导链追溯到它保障的必要条件。

---

## 第一层：成功的定义

> 给定当前状态 A 和期望终态 B，在 AI 的能力和信息范围内，找到从 A 到 B 的最优路径。

"最优"意味着：不仅路径本身正确，而且用户能理解并执行。一个完美的方案如果用户无法理解，等于没有方案。

## 第二层：问题的两种基本类型

所有问题归入两类，或其混合：

| | 收敛型（发现答案） | 发散型（创造答案） |
|---|---|---|
| 本质 | 答案已存在于事实中 | 答案需要被构造 |
| 解空间 | 有限，可枚举 | 无限，需生成 |
| 瓶颈 | 信息完备性 + 推理正确性 | 生成质量 + 评估标准 |
| 失败模式 | 遗漏、推错 | 视野窄、过早收敛 |
| 匹配策略 | 系统枚举 → 排除 → 验证 | 约束识别 → 多候选生成 → 评估选择 |

大部分真实问题是混合型——不同子问题分属不同类型。**用错了策略两边都会失败**：用枚举法做创造性子问题导致视野窄，用发散法做诊断性子问题导致遗漏。

## 第三层：五个必要条件

实践中，从 A 到 B 的最优路径需要以下五类核心条件被满足；这五条覆盖大多数场景但不是封闭清单（见本层末尾的"是否只有这五个？"）：

| # | 必要条件 | 失败时的后果 |
|---|---------|-------------|
| NC1 | **正确理解终态** — 你在解决正确的问题 | 在错误的问题上做完美的分析（Challenger：解的是"如何按时发射"而非"如何安全发射"） |
| NC2 | **正确理解当前状态** — 你知道起点在哪 | 方案建立在错误的前提上（忽略已有系统，从零设计已存在的东西） |
| NC3 | **充分的问题空间信息** — 收敛型需要穷举，发散型需要约束和素材 | 收敛型：遗漏关键因素；发散型：生成的方案不满足未知约束 |
| NC4 | **可靠的推理** — 从信息到结论的过程正确 | 信息齐全但结论错误（确认偏误、过度推断、逻辑跳跃） |
| NC5 | **有效的传达** — 结论被理解且可执行 | 正确的结论无法转化为行动 |

**为什么是这五个？** 从 A→B 路径的关键节点反推：你必须知道 B 在哪（NC1）、A 在哪（NC2）、A 和 B 之间的地形（NC3），然后正确地规划路线（NC4），最后让执行者能看懂地图（NC5）。去掉任何一个，路径就断了。

**为什么不能更少？** 每个条件覆盖独立的失败模式——NC1 失败（理解错问题）和 NC4 失败（推理错误）是完全不同的错误，不能用同一种方法预防。

**是否只有这五个？** 这五类覆盖了实践中最频繁的失败模式，但不是封闭的充要清单。特定问题域可能有额外条件——目标函数的选择（何谓"最优"？）、时间/预算约束、执行者能力边界等。分析具体问题时不以此清单截断搜索，仍从问题本身出发判断还有没有别的失败维度。

## 第四层：方法论——确保每个条件被满足

### 步骤 1：理解与对齐（保障 NC1 + NC2）

确认终态和当前状态。这一步的特殊性：**它的失败模式对自身不可见**——你不知道自己误解了问题（这就是误解的定义）。所以必须通过外部校验（用户确认）来兜底。

### 步骤 2：拆解、分类、收集（保障 NC3）

将问题拆解为子问题，判断每个子问题的类型（收敛/发散），用匹配的策略收集信息：
- **收敛型子问题**：建立枚举框架 → 系统扫描 → 双向交叉核对（通用知识→项目，项目→通用知识）
- **发散型子问题**：识别约束和评估标准 → 收集素材和先例

**拆解方式跟随问题的自然结构。** 每个问题都有自己的自然结构——诊断问题是假设空间，层级系统是树，因果问题是链。选择能让子问题对用户一目了然的拆解方式，并说明理由。

**完备性来自多个独立视角的交叉核对。** 拆完后，不看自己的拆解结果，仅从问题描述出发推导"应该覆盖什么"，然后对比找差异。多 agent 模式下，侦察者的地形图提供第三份独立清单，形成三路交叉验证。

### 步骤 3：推理（保障 NC4）

从信息推导结论。关键区分：
- **确认性推理**（寻找支持自己假说的证据）— 人/模型的本能，但不可靠
- **区分性推理**（寻找能区分不同假说的证据）— 鉴别诊断和 ACH 的方法，更可靠

每个结论需要：证据 → 推理链 → 结论。区分事实（读到的）、推理（推导的）、假设（没有证据的）。

### 步骤 4：合成与呈现（保障 NC5）

将子问题的结论组合为整体方案，检查子解之间的冲突和交互。呈现时先结论后展开——用户在第一层就能判断方向对不对。

### 步骤 5：独立验证（元保障——覆盖所有条件）

每个步骤的失败模式对该步骤自身不可见：
- 步骤 1 的失败（误解问题）→ 用户校验兜底
- 步骤 2 的失败（遗漏信息）→ 双向核对 + 充分性检验兜底
- 步骤 3 的失败（推理错误）→ 自我挑战 + 区分性推理兜底
- 步骤 4 的失败（传达不清）→ 分层呈现 + 用户反馈兜底

当问题的完备性要求超过单 agent 自我验证的可靠度时，需要结构性地引入独立验证（多 agent）——这是两档架构的触发条件。

### 覆盖性论证

| 必要条件 | 保障步骤 | 失败兜底机制 |
|---------|---------|-------------|
| NC1 正确理解终态 | 步骤 1 | 用户校验 |
| NC2 正确理解现状 | 步骤 1 + 步骤 2 | 用户校验 + 系统收集 |
| NC3 充分信息 | 步骤 2 | 枚举框架 + 双向核对 + 充分性检验 |
| NC4 可靠推理 | 步骤 3 | 区分性推理 + 自我挑战 |
| NC5 有效传达 | 步骤 4 | 分层呈现 + 用户反馈 |
| 元保障 | 步骤 5 | 多 agent 独立验证（按需） |

上表确保 NC1-NC5 的每个条件都有对应的保障步骤和失败兜底。若未来识别出第六个失败模式，需要同步补充保障路径——不是靠已有步骤兼容，而是独立设计新保障。

## 第五层：环境约束与实现方法

上述方法论是通用的。在 Claude Code 环境下，四个不可消除的约束决定了具体的实现方式：

| 环境约束 | 对方法论的影响 | 实现方法 |
|---------|--------------|---------|
| **信息不对称** — 人类知道意图，AI 知道代码 | 步骤 1 的失败风险高于通常情况 | 问题模糊时列出多种理解方向让用户选择，不自行猜测 |
| **工具可达** — AI 能 Read/Grep/Bash/WebSearch | 步骤 2 的责任在 AI 而非用户 | 穷尽工具手段后才向用户求助；收集优先级：上下文→代码库→网络→脚本 |
| **注意力有限** — 上下文窗口有限，指令竞争注意力 | 步骤 4 和 skill 自身都受此约束 | 延迟加载（阶段分离）；输出先结论后展开；skill 文本保持精简 |
| **训练倾向性** — 模型有"收到输入就产出答案"的默认行为 | 所有步骤都可能被跳过 | 纪律（见第六层） |

## 第六层：纪律——对抗"知道但不做"

纪律不是方法论的替代品，而是方法论在"模型会知道正确做法但训练倾向让它不这么做"这个事实下的补丁。每条纪律标注它保护的步骤和对抗的倾向：

| 纪律 | 保护 | 对抗的训练倾向 |
|------|------|--------------|
| 没有跟用户对齐问题之前，不做任何分析和结论 | 步骤 1（NC1） | 收到输入就产出答案，用代码里的第一个匹配项替代用户真实意图 |
| 不要把能自己查到的东西推给用户 | 步骤 2（NC3） | 遇到不确定就问用户，而不是自己用工具查 |
| 结论的强度不能超过证据的强度 | 步骤 3（NC4） | 把逻辑上说得通的推断包装成事实 |
| 不要把"找到一些"当成"找到全部" | 步骤 2（NC3） | 5-8 个发现后产生虚假完成感 |
| 不要用结构伪装深度 | 步骤 4（NC5） | 用完整结构填充输出，每个板块都要写点什么 |

## 第七层：两档架构

方法论流程在两档之间不变，变的是验证机制的结构：

**单 agent（默认）**：步骤 1-5 由同一个 agent 执行。深度按问题需要连续调节。自我验证依赖双向核对、充分性检验、用户校验。

**多 agent（按需触发）**：当完备性要求超过单 agent 自我验证的可靠度时启用。结构性引入独立验证——不同 agent 拥有独立的注意力池、可以信息隔离、可以对抗性审查。

触发条件：问题要求"漏一个就失败"级别的完备性（安全审计、迁移评估、合规检查等），且问题空间大到单 agent 的注意力预算不足以同时保持枚举完备性和推理质量。
