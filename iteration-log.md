# First Principles Skill 迭代日志

## [2026-03-20] 迭代 #6

**触发会话**：用户反馈每次拆解问题时都被强制拆成"两个正交轴→矩阵"，产出的轴名抽象且难以理解。用户质疑：(1) 为什么是两个而不是更多？(2) 为什么一定要拆成矩阵？

**观察到的问题**：
1. "两个正交轴"是一个过度具体的实现，被硬编码为通用步骤。理论依据"不止从一个角度看问题"只能支撑"多视角"，推不出"恰好两个轴"和"必须正交"
2. 不是所有问题都有两个天然维度。诊断问题的自然结构是假设空间，层级系统是树，因果分析是链——硬套矩阵会产出空洞的轴名
3. 完备性检查（原 3c 自检缺口）放在收集阶段末尾太晚——拆解定义搜索空间，拆解错了收集补不回来
4. 多 Agent 模式下，侦察者的地形图本可用于验证拆解完备性，但当前流程中这个交叉验证机会被浪费了

**改动**：
- **single-agent.md 步骤 2**：拆解方式跟随问题的自然结构（假设空间、层级树、因果链等），选择时说明理由。拆解后紧跟完备性检查（独立推导→对比→找差异）
- **multi-agent.md 步骤 3**：同上 + 三路交叉验证（拆解 × 地形图 × 独立推导取并集），充分利用侦察者的独立信息源
- **derivation.md**：更新拆解理论——拆解方式跟随问题自然结构，完备性来自多个独立视角的交叉核对
- **phase-2-single/multi.md**：报告数据字段从 `matrix` 改为 `decomposition`（类型化，支持 tree/list/matrix 等）
- **report-template.html**：渲染函数支持多类型分发，兼容旧 matrix 格式

**依据**：
- 用户反馈："你拆的这些正交轴我完全看不懂"
- 用户反馈："为什么你会觉得拆成两个正交轴是分析任何问题的通用方法？"
- 分析：MECE 是质量标准但缺方法，固定矩阵是方法但缺弹性。拆解方式应跟随问题结构，完备性应由独立检查保障

**预期效果**：
- 拆解结构对用户一目了然——因为选择了问题本身的自然结构
- 完备性保障更强——单 agent 有独立推导交叉检查，多 agent 有三路交叉验证
- 多 Agent 模式充分利用侦察者的独立信息源

## [2026-03-16] 迭代 #5

**触发会话**：用户反馈阶段一输出中"信息缺口与假设"作为固定结构块，造成两个问题：(1) 结构性压力迫使 Agent 制造不存在的缺口，延迟本可完成的分析；(2) 即使没有真正的信息缺口，也要填充这个板块，违反了 skill 自身"不要用结构伪装深度"的原则。

**改动**：
- SKILL.md 第五步：将"信息缺口与假设"从固定板块改为条件性板块"需要你帮忙的地方"——只在穷尽自身手段后仍有关键信息无法获取时才出现，没有就跳过
- 出现时要求说清三件事：需要什么信息及其对分析的影响、已尝试的途径、用户可以怎么帮忙
- 将"信息缺口"从第 3 项降为第 4 项（条件性），"拟定的分析方向"升为第 3 项（必出）
- phase-2.md 同步调整：阶段一产出描述从"问题陈述、已掌握信息、信息缺口"改为"问题陈述、已掌握信息、分析方向"

**依据**：
- 用户反馈："很多时候，其实你探索下来的用户问题并不一定有待确认的信息缺口"
- 用户反馈："我现在经常会觉得你在第一轮分析问题时只会进行部分分析，反而为了要留出一些待确认的信息缺口而不去分析完整"
- 与 skill 自身原则的矛盾："不要用结构伪装深度。如果某个环节没什么可说的，跳过它"

**预期效果**：
- 信息充分时直接进入分析，不再人为制造"待确认"环节
- 真正需要用户帮忙时，提问更精准、更可操作（说明需要什么、为什么、怎么帮）
- 阶段一产出更诚实地反映实际认知状态

## [2026-03-16] 迭代 #4

**触发会话**：用户反馈两个问题：(1) 信息收集阶段不够主动，"信息缺口"里列了大量 Agent 自己就能查到的东西；(2) 分析和信息收集完全解耦导致不知道该收集什么信息。

**观察到的问题**：
1. 第四步"信息审计与收集"是盲目扫描——Agent 还没尝试回答问题，不知道需要什么信息
2. "信息缺口"的定义过于宽泛——"我没能确认的部分"包含了大量 Agent 自己能查到的东西
3. 两阶段结构缺少后续交互的指导——用户对初步结论不满意时没有方法论支撑迭代

**改动**：
- SKILL.md 第四步重构为三个子步骤：4a 初步分析（内部思考，暴露信息需求）→ 4b 六维度自检（带着需求扫描盲区）→ 4c 穷尽自身能力收集（四层优先级 + 真伪缺口判断标准）
- SKILL.md 第五步"信息缺口"重定义为"仅限穷尽自身手段后仍无法获取的内容"，要求说明尝试过什么途径
- 两阶段扩展为三阶段：新增阶段三"持续迭代"（phase-2.md 末尾），定义了微循环：识别新信息需求 → 定向补充收集 → 基于新信息重新分析

**依据**：
- 用户反馈："既然这些信息就在上下文里，我自己也提供不了更多，你为什么不自己去确认呢？"
- 用户反馈："如果你没有尝试去分析这个问题，你是不是其实没办法判断要回答这个问题究竟需要哪些完备的信息？"
- 测试对比：新版在 Stage 1 prompt 设计任务上信息缺口分类更清晰（区分了"需要用户提供的"和"需要自己设计的"），额外收集了被审计文档类型特征

**预期效果**：
- 信息收集由初步分析驱动，不再盲目扫描
- "信息缺口"只包含真正需要用户提供的内容
- 用户反馈后有方法论支撑持续迭代，而非每次从头开始

## [2026-03-14] 迭代 #1

**触发会话**：用户反馈在讨论 Ember CLI 方案时，Agent 完全忽略了已有的 Tauri 桌宠生态（代码、下载更新机制、版本管理），把新客户端当全新功能从零设计，导致做出来的东西与现有系统完全脱节。

**观察到的问题**：
1. Agent 在信息不足时没有意识到信息不足，直接开始输出方案（unknown unknowns 问题）
2. 遇到不确定的信息时倾向于直接向用户提问，而用户不一定能回答技术细节——应该自己先去查
3. 原有"信息完备性决定分析质量"只是一条空洞的信念声明，没有转化为可操作的行为纪律

**改动**：
- 核心信念：重写"信息完备性"为"你不知道自己不知道什么"，明确对抗盲区的方法是结构化自检 + 用户验证
- 核心信念：将"能自己验证的就不要推给人类"改写为"探索先行，展示发现"——强调的不只是自己去查，更是把查到的东西透明地展示给用户
- 阶段一：增加"定位请求与现有事物的关系"
- 阶段二：全面重写，分为三个子步骤——信息完备性自检（六维度框架）、探索先行、呈现认知基础
- 六维度框架（现状、目标、动因、约束、关联、历史）作为通用自检工具，适用于任何类型的问题

**依据**：
- 具体案例：Ember CLI 设计时完全忽略了已有的 Tauri 桌宠、下载更新机制、共享协议（conv-852 暴露的 I/O 问题是动因，但 Agent 不知道）
- 用户反馈："他完全没有基于我们整个项目的背景和语境去跟我讨论问题"
- 用户反馈："我经常发现你提的一些技术性问题，我其实根本就不知道答案"

**预期效果**：
- 讨论技术方案时，Agent 会先自检六个维度的信息完备性，发现明显缺口后主动去探索
- Agent 会在给出方案前先呈现"我了解到了什么"，让用户有机会发现遗漏
- 减少向用户提技术问题的频率，改为展示自己的发现让用户补充

## [2026-03-14] 迭代 #1 补充

**触发**：测试 3（上下文工程优化）中，新版 skill 仍然出现推理越界——看到 24 份优化记录就断言主要系统性问题已被修复，但这个结论需要用户满意度数据，模型只有操作记录。

**观察到的问题**：之前的改进全部集中在信息收集侧（收集够不够、呈现了没有），但推理侧缺少纪律——即使信息收集到位，从信息到结论的推导本身也可能越界。模型做了看起来合理的推理，但没有自检这个结论需要什么证据？我有吗？

**改动**：核心信念区新增结论的强度不能超过证据的强度——要求对每个判断自检：支撑它需要什么证据？有还是没有？没有就只能是假设。

**依据**：信息收集完备性和推理严谨性是两个独立的问题。前者回答我知道什么，后者回答我能从中得出什么。两者都需要纪律。

**预期效果**：模型在产出结论时会更审慎地区分证据支撑的结论和逻辑上说得通但缺乏证据的假设，减少把推断包装成事实的情况。

## [2026-03-14] 迭代 #1 补充 2

**触发**：Iteration-2 模糊提示词测试中，新旧两版 skill 都把“文件被锁住改不了”理解为 P4 只读权限问题，而用户本意是 UE 引擎文件锁。Agent 在代码库里找到了 P4 相关代码就直接锁定了这个解释，从未停下来确认用户的真实意图。

**观察到的问题**：阶段一“理解问题”和阶段二“信息收集”被混为一体。Agent 在阶段一就去翻代码库，找到一个“看起来能对上”的解释就直接跑了。用户反馈：“在没对齐需求和用户的问题之前，擅自给出结论和做出分析判断，是绝对错误的行为。”

**改动**：阶段一开头加入硬性纪律——“在没有跟用户对齐问题之前，不做任何分析和结论”。明确要求：当问题描述模糊时，列出多种可能的理解方向让用户选择，而不是自行挑一个开始分析。

**依据**：模糊提示词下，两版 skill 都把问题理解错了。这说明原来的“复述用户的问题，让用户确认或纠正”这条指令力度不够，需要升级为硬性纪律。

**预期效果**：面对模糊提示词时，Agent 会先呈现多种可能的理解方向而不是直接跳入分析，避免浪费时间在错误的问题上做深度分析。

## [2026-03-14] 迭代 #2

**触发会话**：基于 iteration-1 到 iteration-3 的全量测试对比，重构 skill 结构并修复回归问题。

**观察到的问题**：
1. 旧结构（四阶段）把 80% 篇幅放在“怎么分析好”上，“理解问题”只是一个小前奏，强化了模型“收到输入就产出答案”的默认倾向
2. 模糊提示词测试中，新旧两版都把“文件被锁住”理解为 P4 只读而非 UE 引擎锁——“复述用户问题”这条规则写了但不被执行
3. 精确提示词回归时，新结构在信息收集深度上出现退化——不再做网络搜索和外部文档调研

**改动**：
- 重构为两阶段结构：阶段一（理解问题）+ 阶段二（分析与呈现），给予“理解问题”对等的结构权重
- 思维纪律和表达哲学提到最前面，作为贯穿两阶段的共享原则
- 阶段一内部分五步：熟悉环境 → 判断清晰度 → 确认目标 → 信息审计 → 呈现认知基础
- 阶段二新增“外部调研”步骤，修复信息收集深度退化问题
- 修复“具体胜过抽象”的例子（原例违反费曼标准）

**依据**：
- 模糊提示词测试：iteration-3 成功识别模糊性并请求对齐，iteration-2 直接跳入分析
- 精确提示词回归：iteration-3 测试 1 缺少外部调研，方案质量不如 iteration-1
- 用户反馈：“在没对齐需求和用户的问题之前，擅自给出结论和做出分析判断，是绝对错误的行为”

**预期效果**：
- 模糊提示词：先对齐再分析，不在错误的问题上浪费时间
- 精确提示词：保持信息收集深度（包括外部调研），同时信息基础更透明

## [2026-03-14] 迭代 #3

**触发会话**：用户提出将"理解问题"和"分析呈现"两阶段解耦为独立文件，通过延迟加载减少认知干扰。

**观察到的问题**：
1. 阶段二的指令（如何组织结论、金字塔呈现等）在阶段一执行期间就已加载到上下文中，拉着模型过早收敛——这与迭代日志中反复出现的"跳过对齐直接分析"是同一根因
2. 两个阶段的方法论共享一个文件的 token 预算，无法各自深化
3. 迭代时修改一个阶段容易引入另一个阶段的回归

**改动**：
- 将 SKILL.md 拆分为两个文件：`SKILL.md`（共享原则 + 阶段一完整方法论）和 `phase-2.md`（阶段二完整方法论）
- SKILL.md 末尾加入硬规则：阶段一完成且用户确认后，才 Read `phase-2.md` 进入阶段二
- 阶段一"呈现认知基础"步骤强化为结构化检查点：问题陈述 → 已掌握信息 → 信息缺口 → 拟定分析方向，明确请求用户确认
- 共享原则（思维纪律、表达哲学）保留在 SKILL.md 中，两个阶段都能访问

**判断框架**（用于评估是否值得拆分）：
- 认知模式切换：两侧是否需要不同的思维方式？
- 用户校验价值：在边界处停下来，用户有值得校验的东西吗？
- 错误传播成本：这一步搞错了，后面浪费多少？
- 指令隔离收益：后阶段的指令会不会干扰前阶段的执行？
四条都强则拆，一两条弱则合并。结论：B→C（信息收集→分析推理）边界四条全强，是最该拆的；A→B（理解→收集）和 C→D（分析→呈现）边界较弱，不拆。

**预期效果**：
- 阶段一执行时模型上下文中完全没有"如何组织结论"的指令，注意力纯粹聚焦在理解和收集
- 每个阶段的方法论可以独立深化和迭代，不互相干扰
- 用户在阶段边界获得一次明确的校验机会，减少信息不全就分析的问题
