# 单 Agent 模式：阶段二

你已完成阶段一，用户已确认认知基础。回顾问题陈述、子问题分类（收敛/发散）、已掌握信息和分析方向，然后开始。

---

## 步骤 1：推理

对每个子问题，按类型选择策略：

### 收敛型子问题（答案存在于事实中）

1. **生成假说**：基于收集到的证据，列出所有合理的假说/答案
2. **区分性评估**：对每条关键证据，问"它支持哪些假说、排除哪些假说"——寻找能区分假说的证据，而非确认偏好的那个
3. **得出结论**：每个结论标注——证据（读到的）、推理（推导的）、假设（没有证据的）

### 发散型子问题（答案需要被构造）

1. **明确约束和评估标准**：什么不能动、什么必须满足、什么是加分项
2. **生成多个候选方案**：至少两个方向不同的方案，不要只生成一个然后论证可行
3. **逐个评估**：每个方案对照约束和标准打分，标注各自的优势和风险
4. **推荐**：给出推荐及理由，同时说明在什么条件下另一个方案会更好

## 步骤 2：挑战

所有子问题推理完成后，退一步自我挑战：

1. **替代结论**：用同样的证据，能否构建一个不同但同样合理的结论？如果能，当前结论的优势在哪？
2. **反面证据**：有没有被低估的证据与当前结论矛盾？
3. **组合检查**：把所有子结论放在一起看——它们之间有没有矛盾？一个子问题的方案会不会破坏另一个子问题的前提？
4. **过度自信检查**：越"确定"的结论越重点审视——确定感可能来自证据充分，也可能来自没有认真找反面

发现真实问题就修正结论，不是列出来就完。

## 步骤 3：呈现

**读取 `presentation.md`**，按其中的原则与结构向用户呈现。深潜与围猎共享同一套呈现纪律。

---

## 持续迭代

用户可能质疑结论、补充新信息、或引向更深的子问题。每轮反馈触发微循环：

1. **识别新信息** — 用户反馈本身就是新信息。他指出了什么错误？暗示了什么遗漏的维度？
2. **定向补充收集** — 穷尽自身能力去查，不推给用户。定向、轻量，不重做全面扫描。
3. **增量更新结论** — 标注"相比上一轮，什么变了、为什么变了"。

如果反馈触及根本性方向问题（目标本身变了），退回阶段一重新对齐。

---

## 报告生成

分析完成后（用户确认最终结论或会话即将结束时），生成可视化报告。

### 前置：写入分析产出文件

深潜模式的分析在对话中完成（不 spawn agent），但仍需将关键产出写入工作目录的 `.md` 文件，以便报告模板渲染。至少写入：

- `final-report.md`（必须）：报告正文。金字塔结构，面向决策者。
- `findings.md`（推荐）：收集阶段的发现汇总。
- `analysis.md`（推荐）：推理与方案设计。
- `challenge.md`（推荐）：自我挑战的结论。
- `conclusion.md`（推荐）：最终裁决。

文件越多，"分析过程"页签内容越丰富。但 `final-report.md` 是唯一必须的。

### 生成步骤

与围猎模式使用**同一个模板** `report-template.html`，替换相同的占位符。区别仅在于：

- `__METHOD__`：填 `"深潜模式（Single Agent）"`
- `__STAGES_JSON__`：深潜模式的阶段定义：
  ```json
  [
    {"id":"collect","label":"收集","desc":"自底向上 + 自顶向下"},
    {"id":"analysis","label":"分析","desc":"推理与方案设计"},
    {"id":"challenge","label":"挑战","desc":"自我对抗审查"},
    {"id":"conclude","label":"结论","desc":"最终裁决"}
  ]
  ```
  如果某阶段没有对应文件，该阶段在 UI 上显示为空（不报错）。

- `__DOCS_JSON__`：同样扫描工作目录所有 .md（排除 final-report.md），`stage` 推断规则同围猎模式。

- `__LABELS_JSON__`：深潜模式文件较少，直接为每个文件定义标签即可。

- `__SESSIONS_JSON__`：同围猎模式，扫描所有 session，更新所有已有 report.html。

其余步骤（`__TITLE__`、`__DESC__`、`__DATE__`、`__REPORT_HTML__`）与围猎模式完全相同。

告知用户报告路径。
