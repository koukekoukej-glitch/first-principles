# First Principles

第一性原理深度分析 Skill。把复杂决策（架构选型、可行性评估、多方案对比、风险审计）拆成可论证的推理链——每个子问题先匹配收敛 / 发散策略，再用独立子 Agent 交叉验证拆解的完备性，让结论的强度不超过证据的强度。

适用于 [Claude Code](https://docs.anthropic.com/en/docs/claude-code) 及任何支持自定义 System Prompt + 文件读写 + Shell 执行 + 子 Agent 调度的 LLM 框架。

[English](README.en.md)

## 特性

- **两类问题、两套策略** — 收敛型问题（答案在事实中）走"系统枚举 → 排除 → 验证"；发散型问题（答案需被构造）走"约束识别 → 多候选 → 评估选择"。子问题分类错配、两种问题都会失败
- **拆解先于收集** — 拆解定义了搜索空间。漏一个维度、后面收集做得再彻底也补不回来。先拆解、再收集
- **独立子 Agent 验拆解完备性** — Spawn 干净上下文的子 Agent、只传问题描述、让它独立列"应该覆盖哪些方面"。和自己的拆解取并集后才进入收集——LLM 在同一上下文内无法真正"忽略"已写过的内容，自己重推一遍是形式独立、实质同质
- **双向收集对抗注意力锚定** — 每个子问题都走两路：自底向上（从已有材料逐层扫描）+ 自顶向下（从领域知识逐项映射）。两路有系统性互补盲点，并行收集才能互相补漏
- **两种执行模式** — **深潜**：单 Agent 完整走理解→拆解→收集→推理→呈现，适合大多数有复杂度的问题；**围猎**：所有阶段都 spawn 子 Agent 并行 + 三路交叉验证，仅用于"漏一个就失败"级别的完备性需求（迁移评估、安全审计、合规检查）
- **结论强度 ≤ 证据强度** — 逻辑上说得通不等于事实上成立。没有证据就标注为假设、不伪装成结论；依赖未验证假设的结论本身也是假设
- **HTML 报告渲染** — 分析结果输出为带证据卡片、置信度标签、推理链可折叠的交互式 HTML 报告

## 五类失败的命名与防御

设计上把"复杂问题分析容易失败"这件事拆成五类必要条件（NC1–NC5）——每一类都对应一种典型的"看起来在分析、其实在跑偏"。Skill 的每条硬约束都标注它保护的必要条件：

| # | 必要条件 | 失败时的样子 |
|---|---------|-------------|
| NC1 | 正确理解终态 | 在错误的问题上做完美的分析（解的是"如何按时发射"而非"如何安全发射"）|
| NC2 | 正确理解当前状态 | 方案建立在错误的前提上（忽略已有系统、从零设计已存在的东西）|
| NC3 | 信息完备性 | "找到一些"被当成"找到全部"、5-8 个发现后觉得差不多了 |
| NC4 | 推理可靠 | 逻辑链每一步都对、合起来却离题 |
| NC5 | 传达失效 | 结论对的、但用户读不懂、无法执行 |

**总原则**：每个步骤的失败模式对自身不可见。误解问题时你不知道自己误解了、遗漏信息时你不知道自己遗漏了。所以每一步都必须有外部兜底——用户校验、子 Agent 独立推导、自我挑战——不要相信"看起来对了"的感觉。

## 两种模式怎么选

**深潜** — 一个分析师完整走全流程、过程中 spawn 一次子 Agent 做拆解的完备性对照、主体仍是单 Agent 工作。

典型场景："这个 bug 根因是什么"、"这个架构怎么设计"、"A 和 B 哪个方案更好"、"这段代码为什么慢"

**围猎** — 侦察、收集、分析、对抗审查各阶段均 spawn 子 Agent、交叉验证、地毯式覆盖。

典型场景："把 X 系统从 A 迁移到 B、不能遗漏任何依赖"、"审查代码有没有安全漏洞"、"评估这个重构会影响哪些下游"

**不确定时选深潜**——深潜过程中发现完备性压力大可以升级到围猎、围猎成本高不能降级。

## 工作流程

```
阶段一：理解问题
  ├─ 步骤 1 理解与对齐         (NC1 + NC2)
  ├─ 步骤 2 拆解与分类         (NC3 前置)
  │   ├─ 用问题的自然结构拆解（假设空间 / 矩阵 / 树 / 因果链）
  │   ├─ 独立子 Agent 推导对比（硬性关卡）
  │   └─ 每个子问题归类（收敛 / 发散）
  ├─ 步骤 3 收集               (NC3)
  │   ├─ 自底向上：从材料逐层扫描
  │   ├─ 自顶向下：从领域知识逐项映射
  │   └─ 自检缺口（复用 step 2 的独立清单）
  └─ 步骤 4 呈现认知基础、请求用户确认

阶段二：分析与呈现
  ├─ 推理（NC4）
  ├─ 对抗审查（NC4）
  ├─ HTML 报告渲染（NC5）
  └─ 迭代
```

阶段一和阶段二**分步加载、分步执行**——在阶段一完成并获得用户确认之前、不读取阶段二文件、不做任何结论。这道门控本身就是 NC1 的保护。

## 安装

```bash
git clone https://github.com/koukekoukej-glitch/first-principles.git ~/.claude/skills/first-principles
```

Claude Code 自动识别触发条件。也可将 `SKILL.md` 作为 System Prompt 加载到其他框架。

### 触发

自然语言触发：

```
> 怎么设计这个模块
> A 和 B 哪个方案更好
> 这个架构有什么风险
> 拆解一下这个问题
> 评估一下可行性
> /first-principles
```

适用判据——问题至少满足以下之一才触发：① 需要拆解成多个子问题 ② 需要先收集信息才能回答 ③ 多方案对比决策 ④ 需要论证链而非单一答案。若问题能 1-2 步直接回答（查个函数、改个字段）不触发。

## 设计原则

1. **拆解先于收集** — 拆解错了、收集补不回来
2. **每一步都要外部兜底** — 自身的失败模式对自身不可见
3. **独立性来自隔离的注意力池** — 自己重推一遍不是独立推导、必须用干净上下文的子 Agent
4. **没有证据就不是结论** — 标注为假设、不伪装成结论
5. **不为填模板而捏造问题** — 找不到需要用户帮忙的事就跳过这个模块、不要为了"显得严谨"硬塞内容
